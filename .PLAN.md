# Why convo-tools Ingestion Is Slow vs mempalace

## Root Causes (ordered by impact)

### 1. Heavy NLP Pipeline (BIGGEST bottleneck)

convo-tools runs **three expensive NLP passes** on every message that mempalace doesn't use at all:

| Step | convo-tools | mempalace |
|------|-------------|-----------|
| Language detection | `langdetect` on every message (`_extract.py:66`) | None — accepts all text |
| Named Entity Recognition | spaCy NER per message in batches of 16 (`_builder.py:58-133`) | None |
| TF-IDF keywords | sklearn TfidfVectorizer over all messages (`_builder.py:227-265`) | Keyword scoring by simple string count (`miner.py:363-368`) |

spaCy alone can take **minutes on 10K+ messages**. langdetect is called per-message (`_extract.py:66`), which is O(n) with a non-trivial model load per language.

### 2. Individual DB Writes vs Batched Upserts

**convo-tools** (`_graph_db.py:155-189`): `upsert_node()` does `SELECT` + conditional `UPDATE`/`INSERT` — **one SQLite round-trip per node**. For 10K messages + entities, that's 20K+ individual DB operations.

**mempalace** (`miner.py:880-904`): Batches chunks in groups of 1000 (`DRAWER_UPSERT_BATCH_SIZE=1000`) and calls `collection.upsert()` once per batch. One forward pass per batch.

### 3. Expensive Embedding Computation

**convo-tools** (`_embeddings.py:66-101`):
- Builds a sparse entity incidence matrix
- Runs **full SVD** (via `scipy.sparse.linalg.svds`) — O(n³) in worst case
- Or node2vec: random walks + co-occurrence matrix + SVD

**mempalace** (`embedding.py`):
- Uses pre-built **ONNX MiniLM-L6-v2** (22M params, 384-dim)
- Single forward pass per batch
- Hardware-accelerated (CUDA/CoreML/DirectML)
- Cached per-process — only loaded once

### 4. Two-Phase Pipeline

convo-tools requires **4 separate CLI commands**:
```
extract → pickle → graph → ingest (kuzu)
```

mempalace does it in **one pass**: `mempalace mine <dir>` handles normalize → chunk → upsert.

### 5. Storage Overhead

convo-tools builds a full knowledge graph: node table, 5 edge tables, entity_int mapping table, keyword edges, co-occurrence edges, embeddings table. All in SQLite.

mempalace stores chunks + metadata in ChromaDB (vector DB with built-in HNSW index).

### 6. No Efficient Incremental Skip

convo-tools tracks `processed_message` IDs (`_graph_db.py:410-426`) but still **loads all messages from pickle** every time (`_builder.py:287-288`).

mempalace has three-layer skip: `file_already_mined()` + `ContentHashDB` + mtime check (`miner.py:822-827`).

## Proposed Optimizations

### Quick Wins (low effort, high impact)

#### QW1: Make NLP Steps Optional
- **File**: `src/convo_tools/_builder.py:202-265`
- **Change**: Add `--skip-ner` and `--skip-keywords` flags. Make spaCy NER and TF-IDF extraction opt-in rather than default.
- **Impact**: Eliminates the two biggest bottlenecks for users who don't need entity extraction.

#### QW2: Batch SQLite Writes in upsert_node
- **File**: `src/convo_tools/_graph_db.py:436-445` (add_nodes_batch exists but is rarely used)
- **Change**: Rewrite `_builder.py:180-196` to collect all nodes/edges first, then use `executemany()` in a single transaction. Currently calls `upsert_node()` per message in a loop.
- **Impact**: 10-50x fewer DB round-trips.

#### QW3: Batch langdetect
- **File**: `src/convo_tools/_extract.py:66`
- **Change**: Instead of `detect_lang()` per message, detect language once per conversation (use the longest message or combine first N messages). Or make it configurable with a `--lang-all` flag to skip detection.
- **Impact**: langdetect is ~5ms per call × 10K messages = 50 seconds saved.

### Deeper Refactors (more effort, structural improvement)

#### DR1: Single-Pass Pipeline
- **Files**: `src/convo_tools/__main__.py` (full command), `_extract.py`, `_builder.py`
- **Change**: Create a `convo-tools ingest-all` command that does extract + graph + embeddings in one pass, like `mempalace mine`. Skip the pickle intermediate.
- **Impact**: Eliminates pickle serialization/deserialization overhead and simplifies the workflow.

#### DR2: Replace Spectral Embeddings with ONNX Model
- **File**: `src/convo_tools/_embeddings.py`
- **Change**: Add an ONNX embedding option using `all-MiniLM-L6-v2` (same as mempalace). Keep spectral as fallback. The ONNX model gives better semantic similarity for a fraction of the SVD cost.
- **Impact**: Faster embeddings + better quality.

#### DR3: Add Content-Hash Incremental Skip
- **File**: `src/convo_tools/_extract.py`
- **Change**: Add a content hash DB (like mempalace's `ContentHashDB`) so already-processed files are skipped even before loading JSON.
- **Impact**: Re-runs on large datasets drop from minutes to seconds.
